## 不可动摇的约束

以下为项目级别的硬性约束，AI 不得违反。接入项目后根据实际架构填写。

1. **数据模型唯一来源**：所有类型/接口定义必须集中在 `[类型定义文件/目录]`，其他文件只能引用，不得重新定义
2. **API 调用唯一入口**：所有外部请求必须通过 `[API 封装文件]` 中已有的函数，禁止在各处自行创建请求实例
3. **UI 组件来源**：优先使用 `[组件目录]` 下的已有组件，新建组件必须放在该目录
4. **环境变量**：仅通过 `[环境变量访问方式]` 访问，所有可用变量定义在 `[环境变量示例文件]`
5. **修改前检查**：任何修改必须遵循 SKILL.md 工作流（任务日志 → 影响分析 → 确认 → 执行 → 验证 → 提交）。提交前必须通过 `bash vibe-control/scripts/check.sh` 全部检查项。
6. **提交规范**：使用 Conventional Commits 格式 `feat/fix/chore/docs: 简短描述`
7. **依赖引入管控**：不得随意引入新依赖（npm/pip/cargo 等）；如需新增，必须在任务描述中说明理由并经过确认。优先复用已安装的依赖实现功能。
8. **秘钥/凭证防护**：API Key、密码、Token、私钥等敏感信息**绝对禁止**写入代码、注释、日志或提交到仓库。统一通过环境变量访问，敏感文件写入 `.gitignore`。
9. **死代码清理**：修改后必须删除被替换的旧代码，**禁止**留注释掉的代码作为"备用"。被替换的逻辑通过 Git 历史追溯，不在源码中保留。
10. **测试覆盖底线**：新增代码的核心逻辑必须有单元测试。修改已有逻辑时，受影响的测试用例必须同步更新。`[测试覆盖命令]` 覆盖率不得低于 `[覆盖率阈值]%`。
11. **项目边界约束**：AI 不得在未通过 `question` 工具确认的情况下，访问或分析任何位于当前 Git 仓库根目录之外的文件（`git rev-parse --show-toplevel` 之外的所有路径）。如需访问其他项目或外部文件，必须先弹出 question 让用户授权。违反此约束视为严重违规。

> 根据项目需要增删约束项。每条约束必须明确可执行，而非泛泛而谈。

## 修改协议

涉及数据模型或 API 契约的变更（接口签名、数据库表结构、环境变量），必须严格按以下四步执行：

### 第1步：影响分析（只读，不写代码）
扫描项目，列出所有受影响的文件和符号。格式：
```
受影响文件 | 涉及符号 | 影响类型（直接/间接）
```
确认后方可进入下一步。

### 第2步：修改计划
为每个受影响文件提出最小修改方案，说明：改哪里、为什么必须改、如何保持向后兼容。

### 第3步：执行与核对
逐文件修改，每完成一个文件更新核对清单：
```
[已完成] 文件名 | 修改内容 | 编译通过 | 关联测试通过
```

### 第4步：回归验证
- 运行项目已有测试套件
- 对修改模块生成新的测试用例
- 手动验证核心路径

## 回滚协议

当修改引入问题需要撤销时，按以下优先级处理：

1. **未提交**：通过 `git checkout` 或 IDE `undo` 逐文件回滚
2. **已提交未推送**：`git revert <commit>` 生成反向提交，保留历史
3. **已推送**：`git revert <commit>` 后推送，如涉及多个提交则逐个 revert
4. **回滚后验证**：回滚完成后必须运行完整测试套件和质量门禁

> AI 在修改出问题时，必须主动告知开发者具体的回滚命令，不得尝试"快速再改一次"来掩盖问题。

## 异常处理

当操作失败（非代码逻辑错误）时，按以下协议处理：

### 推送失败

1. **被拒绝（远程有新提交）**：`git pull --rebase` → 解决冲突 → `git push`
2. **网络/权限错误**：检查网络连接和仓库权限，重试 `git push`
3. **pre-push 钩子拦截**：查看检查输出 → 修复问题 → 重新 push

### 拉取/合并冲突

1. `git status` 查看冲突文件
2. 手动解决冲突标记（`<<<<<<<` / `>>>>>>>`）
3. `git add <冲突文件>` → `git rebase --continue`（或 `git merge --continue`）

### 注入失败（inject.sh 中途退出）

1. 运行 `bash vibe-control/scripts/recover.sh` 诊断
2. 根据输出修复缺失项
3. 或重新运行 `bash vibe-control/scripts/inject.sh` 完整注入

### 钩子损坏

1. 运行 `bash vibe-control/scripts/recover.sh` 诊断钩子状态
2. 重新安装：`cp vibe-control/scripts/pre-commit .git/hooks/ && cp vibe-control/scripts/pre-push .git/hooks/`
3. 确认权限：`chmod +x .git/hooks/pre-commit .git/hooks/pre-push`

### 控制文件损坏（.vibe/ 被意外删除）

1. 运行 `bash vibe-control/scripts/recover.sh` 诊断
2. 恢复：`bash vibe-control/scripts/inject.sh`（不会覆盖已有 .gitignore 配置）

> AI 遇到上述任何异常时，必须主动输出具体恢复命令，不得含糊说"出错了请手动处理"。

## 关联模块声明

> 声明项目中存在强耦合关系的模块对，修改其中一方时必须联动检查另一方。

格式示例：
```
- **[模块A]** ↔ **[模块B]**：[说明关联关系及联动检查要点]
```

常见关联模式：
- 数据模型 ↔ 所有消费方（页面、组件、API、数据库查询）
- API 接口 ↔ 所有调用方（前端页面、Hooks、其他服务）
- 数据库表结构 ↔ ORM 模型、数据迁移脚本
- 共享组件 ↔ 所有引用该组件的页面

> 项目特定关联关系详见 `DEPENDENCY_MAP.md`

## 质量门禁

以下条件全部满足才能提交。接入项目后替换为项目实际检查命令。

1. `[类型检查命令]` 零错误
2. `[Lint 命令]` 零警告
3. `[测试命令]` 全部通过，覆盖率达到 `[覆盖率阈值]%` 以上
4. 新增代码有对应的单元测试
5. 敏感信息扫描通过（`bash vibe-control/scripts/check.sh` 中已包含）
6. 按需：`[构建命令]` 成功、`[性能检查命令]` 无劣化
7. 新增脚本/模块后，必须更新 `README.md` 文件说明表，确保文档与实现同步
8. 所有修改/新增的文件必须出现在任务日志的「影响分析」表中，`check.sh` 会做对齐检测，遗漏即拦截

> 如项目暂无某项检查，可标注"暂未配置"，待后续接入时补充。
