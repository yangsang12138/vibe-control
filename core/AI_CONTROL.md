## 项目类型定义
| 类型 | 特征 | 推荐架构 |

## 不可动摇的约束
1. **数据模型唯一来源**：所有类型定义必须集中在 `types/models.ts` 或等价文件中，其他文件只能引用，不得重新定义
2. **API 调用唯一入口**：所有外部请求必须通过 `lib/api.ts`（或等价文件）中已封装的函数，禁止在各处自行创建 fetch/axios 实例
3. **UI 组件来源**：优先使用 `components/ui/` 目录下的已有组件，新建组件必须放在该目录
4. **环境变量**：仅通过 `process.env` 访问，所有可用变量定义在 `.env.example`
5. **修改前检查**：任何修改后必须通过 TypeScript 类型检查（`npx tsc --noEmit`）和 ESLint（`npm run lint`）
6. **提交规范**：使用 Conventional Commits 格式 `feat/fix/chore/docs: 简短描述`

## 修改协议
涉及数据模型或 API 契约的变更（接口签名、数据库表结构、环境变量），必须严格按以下四步执行：

### 第1步：影响分析（只读，不写代码）
扫描项目，列出所有受影响的文件和符号。格式：
```
受影响文件 | 涉及符号 | 影响类型（直接/间接）
```
确认后方可进入下一步。

### 第2步：修改计划
为每个受影响文件提出最小修改方案，说明：改哪里、为什么必须改、如何保持向后兼容。

### 第3步：执行与核对
逐文件修改，每完成一个文件更新核对清单：
```
[已完成] 文件名 | 修改内容 | 编译通过 | 关联测试通过
```

### 第4步：回归验证
- 运行项目已有测试套件
- 对修改模块生成新的测试用例
- 手动验证核心路径

## 关联模块声明
- **用户模块** ↔ **鉴权模块**：修改用户数据模型、角色权限，必须联动检查鉴权中间件和 Token 生成逻辑
- **数据模型** ↔ **所有消费方**：修改 types 定义，必须联动检查所有引用该类型的组件、API、数据库查询
- **API 接口** ↔ **调用方**：修改接口签名，必须联动检查所有调用该接口的前端代码
- 项目特定关联关系见 `DEPENDENCY_MAP.md`

## 质量门禁
以下条件全部满足才能提交：
1. `npx tsc --noEmit` 零错误
2. `npm run lint` 零警告
3. 新增代码有对应的单元测试
4. 已有测试全部通过
